# M4：本地多任务物理隔离与受控并发

## 唯一目标

M4 通过 Git Worktree 为单机上的多个任务提供独立代码目录、工作分支、运行目录和生命周期，并用本地 FIFO 调度器限制并发。Purpose → Brainstorming → Plan → Verify → Archive 意图环和 Plan → Dev → Test → Refactor → Plan 执行环保持不变，Plan 仍是共享桥梁。

## 与 M3 的关系

M4 在任务启动前分配 Worktree，在任务生命周期中维护外部调度记录，并复用 `M3Runner`。Checkpoint、RunContext、Change Spec、反馈路由、安全恢复、版本化 Spec/Plan/Verify 和 Archive 仍完全由 M3 负责。

`TaskRecord` 是唯一新增模型。它是双环之外的任务级支持记录，而不是双环核心业务模型；它只回答任务使用哪个仓库、Worktree、分支和运行目录，以及当前调度状态。

不建立 `WorktreeRecord`，因为 Worktree 的路径、分支和基线已经直接属于 `TaskRecord`，拆分会产生重复身份。不建立 `SchedulerState`，因为运行数、可用槽位和 FIFO 队列都能从注册表中的 `TaskRecord` 实时计算，持久化派生状态会引入一致性风险。

## 隔离与状态规则

- 分支固定为 `sdd/<task-id>`，目录固定为 `<worktrees_root>/<task-id>`。
- 一个 task_id、run_id、分支和 Worktree 路径只能属于一个任务。
- 创建不切换主仓库分支；不 reset、clean、merge、push 或自动删除分支。
- `queued` 按 `created_at` 和注册顺序 FIFO 获取槽位；默认最多两个 `starting`/`running` 任务。
- `awaiting_human` 和 `blocked` 释放槽位；恢复只回到 `queued`。
- `completed`、`failed`、`cancelled` 是终态。
- 清理只移除已验证、干净且允许清理的终态 Worktree，保留分支、归档和 TaskRecord。

## 明确不做

M4 不提供分布式 Worker、消息队列、Redis、数据库、Web 管理端、多 Agent、多模型或 Registry 系统、多仓库协作、PR/merge/push/deploy 自动化、企业权限、多租户、跨机器恢复、长期记忆或知识图谱。

## 完成标准

唯一模型、Worktree 管理、严格 YAML 注册表、本地锁、并发上限、FIFO、暂停/恢复/取消/清理、M3Runner 复用和三任务验收均有测试；M1–M3 全部回归通过；使用手册与快速开始的 API 示例可执行；`progress.yaml` 无 pending 项。
