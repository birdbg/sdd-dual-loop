# M1：双环最小可运行闭环

## 目标

M1 只验证双环架构能够以最小范围完成一次端到端运行。

## 最短路径

```text
输入开发要求
→ Purpose
→ Brainstorming
→ Spec + Plan
→ Verify
→ Dev
→ Test
→ Refactor
→ Archive
```

## 范围限制

| 维度 | M1 限制 |
| --- | --- |
| 技术栈 | Python + FastAPI |
| 任务规模 | 一个小功能 |
| 代码库 | 单仓库 |
| 执行方式 | 顺序执行 |
| 模型数量 | 一个 |
| 测试范围 | 单元测试 |
| 返工次数 | 最多一次 |
| 存储方式 | 本地 Markdown / YAML |
| 人工门禁 | Verify 后由人工确认是否进入 Dev |

## 暂不实施

- 多 Agent 并行
- 多模型路由
- Skill 注册中心
- Web 页面
- Git 自动合并
- 多项目
- 权限
- 企业治理
- API、IT、E2E 全套测试
- 自动 Change Spec

以上项目均不属于 M1，不应在 M1 实施过程中提前引入。

## 实施顺序

M1 不再拆分多个版本，只按以下任务顺序推进：

1. 目录和 RunContext
2. Purpose
3. Brainstorming
4. Spec + Plan
5. Verify
6. Dev
7. Test
8. Refactor
9. Archive
10. 端到端案例

每个任务完成时，必须形成一次代码提交或在 `progress.yaml` 中记录一个包含验证证据的明确检查点。前一任务未完成检查点时，不进入后一任务。

## M1 完成标准

M1 只有在一次真实端到端运行中完成以下全部步骤，才可将状态标记为 `completed`：

```text
输入一个小型 FastAPI 需求
→ 形成 Purpose
→ 完成至少两个方案的 Brainstorming
→ 形成可测试 Spec
→ 形成 Task Plan
→ Verify 通过
→ 修改真实代码
→ 运行真实单元测试
→ 完成一次必要的 Refactor
→ 再次测试通过
→ 生成 Archive
```

未执行真实代码修改、未运行真实单元测试、未进行 Refactor 后复测，或归档不完整，均不得判定 M1 完成。

### 归档最小目录

每次完整运行必须在 `runs/<run_id>/` 下至少生成：

```text
runs/<run_id>/
├── input.md
├── purpose.md
├── brainstorming.md
├── spec.yaml
├── plan.yaml
├── verify.yaml
├── code-diff.patch
├── test-result.yaml
├── refactor.md
└── archive.md
```

以上文件必须来自真实运行，不得为通过完成检查而预先伪造。
