# M2：未知 FastAPI 仓库的最小自主闭环

## 定位

M2 在保持双环架构不变的前提下，将 M1 的固定示例执行提升为面向未知目录结构的小型 Python/FastAPI 仓库执行。

```text
输入一个小型 FastAPI 仓库和开发要求
→ 自动了解仓库
→ 识别相关代码
→ 形成 Spec 和 Plan
→ 在隔离分支修改
→ 自动选择并运行测试
→ 测试失败后有限返工
→ 输出 Diff 和归档
```

M2 只解决五个问题：仓库理解、隔离工作区、真实开发工具、测试失败返工和最小 CI。除此之外的通用化、平台化和企业化能力均不属于 M2。

“未知仓库”是指框架事先不知道目标仓库的源码目录、模块名、FastAPI 入口、测试路径和测试命令。M2 不承诺理解任意语言、任意规模或任意 Python Web 框架。

## 双环架构不变

M2 不新增、删除、合并或重排核心节点：

```text
Purpose
→ Brainstorming
→ Plan
→ Verify
→ Dev
→ Test
→ Refactor
→ Archive
```

反馈方向继续遵循架构总纲：

```text
实现问题 → Dev / Test / Refactor
计划问题 → Plan
规格问题 → 人工 Change Spec → Brainstorming → Plan → Verify
目标问题 → Purpose
```

仓库扫描、文件工具、Git 分支和 CI 是双环的支撑能力，不是新的架构节点。M2 不以自动化为由绕过 Purpose、Spec、Plan 或 Verify。

## 与 M1 的差异

| 维度 | M1 | M2 |
| --- | --- | --- |
| 目标仓库 | 固定示例 | 输入任意符合边界的小型 FastAPI 仓库 |
| 代码路径 | 固定路径 | 运行时识别 |
| 测试命令 | 固定命令 | 从仓库证据中发现 |
| 开发方式 | 直接修改示例 | 独立工作分支修改 |
| 工具 | 面向固定案例 | 仓库内最小文件工具集 |
| 返工 | 固定路径的一次验证 | 测试失败后分类并最多返工两次 |
| 执行组织 | 人工组织 | 双环内自动推进，关键越界由人工决策 |

## 支持边界

M2 仅支持同时满足以下条件的目标仓库：

- 单一 Git 仓库；
- 主要语言为 Python，Web 框架为 FastAPI；
- 可在单机工作区内理解和测试的小型项目；
- 需求是一个边界明确的小功能或小型缺陷修复；
- 依赖信息和测试入口可从仓库文件或常见 Python 约定中确定；
- 不依赖未授权的生产系统、真实密钥或不可获得的外部服务。

框架必须根据实际扫描结果判断仓库是否符合边界。无法确认时应停止并说明原因，不得靠固定路径或猜测继续修改。

## 问题一：仓库理解

### 目标

M1 已知目标文件位置，M2 必须在没有预设路径的情况下形成满足当前任务所需的仓库画像。

### 必须识别

- 项目目录结构；
- Python 包位置；
- FastAPI 应用入口；
- 路由文件；
- 与需求相关的业务代码；
- 测试目录及相关测试；
- 依赖文件；
- 可执行的测试命令。

识别结果必须来自仓库证据，例如目录结构、Python 导入、FastAPI 实例和路由注册、`pyproject.toml`、`requirements*.txt`、README、测试配置或 CI 文件。框架应区分已确认事实和仍待验证的推断。

### 最小产物

```yaml
repository_profile:
  language: python
  framework: fastapi
  source_roots:
    - src
  test_roots:
    - tests
  entrypoints:
    - src/app/main.py
  route_files:
    - src/app/routes/users.py
  relevant_files:
    - src/app/services/users.py
  dependency_file: pyproject.toml
  test_command: python -m pytest -q
  evidence:
    - pyproject.toml
    - src/app/main.py
    - tests/conftest.py
```

字段允许为空或有多个值，但必须说明未识别原因。仓库画像面向本次任务，不建设通用全语言代码理解、知识图谱或完整仓库语义模型。

### 对双环的作用

- Brainstorming 使用仓库画像比较实现方案并形成 Spec；
- Plan 将 Spec 映射到实际文件、符号和测试；
- Verify 检查路径、影响面和测试命令是否有证据支持；
- Dev 和 Test 只在通过 Verify 的范围内执行。

## 问题二：隔离工作区

### 目标

代码修改不得直接发生在默认分支。M2 第一版采用独立 Git Branch，不要求 Git Worktree。

```text
主仓库
→ 确认基线和工作树状态
→ 创建工作分支
→ 在工作分支修改
→ 生成相对基线的 Diff
→ 人工决定是否合并
```

### 必须满足

- 记录目标仓库、默认分支、基线提交和工作分支名；
- 创建分支前检查未提交变更，禁止覆盖或混入用户已有工作；
- 不直接修改 `main`、`master` 或识别出的默认分支；
- 不自动合并；
- 不自动推送；
- 所有代码改动都能通过相对基线的 Diff 查看；
- 失败时能够回到运行前基线，且不损坏用户原有状态。

“可回退”必须通过保留基线和原始内容实现，不等于运行失败后擅自执行破坏性 Git 命令。无法安全创建隔离分支时，应停止并归档原因。

### 最小分支记录

```yaml
workspace:
  repository: /path/to/repository
  base_branch: main
  base_commit: <commit-sha>
  work_branch: sdd/<run-id>
  initial_worktree_clean: true
  merge: manual
  push: disabled
```

## 问题三：真实开发工具

### 目标

Dev 不再只操作固定示例文件，而是通过一组受限、可审计的真实文件工具完成 Plan。

### 最小工具集

| 工具 | 职责 |
| --- | --- |
| `list_files` | 列出仓库内文件和目录 |
| `read_file` | 读取仓库内文本文件 |
| `search_text` | 在仓库内搜索文本或符号 |
| `write_file` | 修改已有文件 |
| `create_file` | 创建新文件 |
| `show_diff` | 查看相对基线的当前改动 |

M2 不建设复杂代码平台、通用 IDE、远程执行平台或任意 Shell 接口。

### 工具约束

- 所有路径必须先规范化，并验证位于目标仓库根目录内；
- 禁止通过绝对路径、`..`、符号链接或其他方式访问仓库外路径；
- 修改已有文件前保存原始内容或足以恢复原文的补丁；
- 每次创建或修改都记录文件路径、操作类型和所属迭代；
- `write_file` 与 `create_file` 不提供隐式批量删除能力；
- 禁止自动删除大量文件；
- 禁止执行危险 Shell 命令；
- `show_diff` 必须以记录的基线为比较对象；
- 实际修改不得超出已验证 Plan，发现需要扩大范围时返回 Plan。

## 问题四：测试失败返工

### 目标

M2 必须真实执行执行环，而不是只验证一条固定成功路径。

```text
Dev
→ Test
→ 失败
→ 分析和分类错误
→ Dev / Test / Plan / 人工 Change Spec
→ 再次 Test
```

### 迭代限制

```yaml
iteration: 0
max_iterations: 2
```

`iteration: 0` 表示首次实现和测试；测试失败后最多允许两次返工。每次返工必须记录触发证据、失败分类、处理决定、代码或测试变化以及复测结果。超过上限仍未通过时，运行以 `failed` 结束，不得宣称完成。

### 失败分类与回流

| 失败类型 | 判断示例 | 回流目标 |
| --- | --- | --- |
| 代码错误 | 实现逻辑、语法、导入或接口行为错误 | Dev；必要时经过 Refactor，再次 Test |
| 测试代码错误 | 新增测试本身错误、测试夹具或测试调用方式错误 | Test；修正测试后再次 Test |
| 计划遗漏 | 遗漏真实影响面、修改步骤或回归范围 | Plan；重新 Verify 后进入 Dev |
| 规格不清 | 验收行为无法唯一确定或与仓库事实冲突 | 请求人工 Change Spec |

M2 可以使用确定性规则结合单一 LLM 进行分类，不建设复杂多 Agent。规格不清时不得自动改写 Spec，必须暂停并请求人工决定；人工确认后再依照意图环重新形成 Plan 和 Verify。

环境缺失、权限不足、密钥缺失或外部服务不可用不应伪装成代码错误。此类情况以 `blocked` 结束或等待人工解除阻塞。

### 测试选择与证据

- 从仓库画像、项目配置、README、CI、测试目录和相关代码中选择命令；
- 优先运行与变更相关的最小测试集，通过后再运行可承受的项目测试；
- 记录命令、工作目录、选择理由、退出码和关键输出；
- “没有找到测试”“测试未执行”和“环境执行失败”均不算通过；
- 禁止通过删除测试、放宽断言或隐藏错误获得通过结果。

## 问题五：最小 CI

### 定位

最小 CI 是 M2 的增强项，用于保证框架本身通过 PR 交付时自动运行测试。它不是未知仓库开发闭环成立的前置能力，也不负责修改目标仓库的 CI。

建议增加 `.github/workflows/test.yml`：

```yaml
name: Tests

on:
  push:
  pull_request:

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"
      - name: Install
        run: |
          python -m pip install --upgrade pip
          pip install -e ".[test]"
      - name: Run tests
        run: python -m pytest -q
```

实施前必须确认 `pyproject.toml` 提供可用的 `test` extra；若项目实际安装方式不同，应调整 Install 步骤，而不是照抄示例。

M2 CI 不包含自动发布、自动部署、Docker、多操作系统矩阵、性能测试、安全扫描或复杂质量门禁。

## 输入与输出

### 最小输入

- 目标 Git 仓库路径；
- 一项自然语言开发要求；
- 可选约束，例如禁止修改的路径或允许执行的命令。

调用方不需要提供固定源码路径、测试路径或测试命令。

### 最小归档

每次运行应在框架自己的 `runs/<run_id>/` 下生成：

```text
runs/<run_id>/
├── input.md
├── purpose.md
├── repository-profile.yaml
├── brainstorming.md
├── spec.yaml
├── plan.yaml
├── verify.yaml
├── workspace.yaml
├── tool-operations.yaml
├── code-diff.patch
├── test-result.yaml
├── feedback.yaml
├── refactor.md
└── archive.md
```

归档不得写入目标仓库。失败和阻塞运行也必须归档真实证据。

最终状态至少包括：

- `completed`：Spec 验收项和约定测试全部通过；
- `failed`：两次返工后仍未通过；
- `blocked`：因环境、权限或必要人工决策无法继续；
- `rejected`：Verify 判定任务超出 M2 支持边界。

只有 `completed` 表示开发要求已经完成。

## M2 完成标准

M2 只有在真实端到端验收中同时证明以下内容，才可标记为 `completed`：

1. 对至少两个未硬编码结构、路径和测试命令的小型 FastAPI 仓库生成正确的 `repository_profile`；
2. 至少覆盖两种源码布局或依赖配置方式；
3. 每次运行都从已记录基线创建独立分支，未直接修改默认分支；
4. Dev 仅通过受限工具集操作仓库内文件，并留下逐文件记录；
5. 自动选择并运行真实测试，输出命令、理由和结果；
6. 至少一个案例由真实测试失败触发返工，并在不超过两次返工后复测通过；
7. 至少一个分类测试证明计划遗漏返回 Plan，规格不清请求人工 Change Spec；
8. 输出的 `code-diff.patch` 与基线及最终工作区一致；
9. 成功、失败或阻塞运行都能生成状态与证据一致的完整归档；
10. 如实施最小 CI，GitHub Actions 能在 Python 3.11 上安装项目并运行测试。

任一核心案例依赖固定仓库名、固定源码路径、固定测试路径或固定测试命令，未建立隔离分支，未运行真实测试，绕过返工分类，或归档缺少真实证据，均不得判定 M2 完成。第 10 项为增强项，不阻塞前九项所定义的 M2 核心闭环验收。

## M2 明确不做

- 多 Agent 并行；
- 多模型自动路由；
- Prompt Registry；
- Skill 市场；
- Web 管理页面；
- 企业权限；
- 多租户；
- 自动合并 PR；
- 自动推送；
- Java、Go、前端等多技术栈；
- 自动生成完整系统；
- 自动部署；
- 自动 Change Spec；
- 知识图谱；
- 长期记忆平台；
- Git Worktree；
- 复杂代码平台；
- 无限制返工。

这些能力属于后续 M3、M4 或企业化阶段，不应在 M2 实施中提前引入。

## 后续实施顺序

本文件只定义 M2，不立即启动编码。后续建议严格按五个问题推进：

1. 仓库理解与 `repository_profile`；
2. 独立 Branch 与基线 Diff；
3. 受限的真实开发工具；
4. 测试选择、失败分类与两次返工；
5. 最小 CI 增强项；
6. 端到端验收和归档检查。
